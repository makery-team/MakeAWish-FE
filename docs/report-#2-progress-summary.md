본 문서는 프론트엔드 아키텍처 개편 작업 완료 후 현재 프로젝트의 상태와, 다음 단계로 나아가기 위해 해결해야 할 블로커(Blocker)를 명확히 정리한 보고서입니다.

## 1. 완료된 작업 (Done) 🚀

### ✅ 프론트엔드 아키텍처 다이어트 (Backend Orchestrator 전환)
- **이전 상태**: 프론트엔드가 AI 서버, 포트폴리오 API, 스토어 API를 각각 찔러서 순서대로 조합하는 `Frontend Orchestrator` 패턴이었습니다. 이로 인해 `storeId`가 누락되면 다음 단계를 진행하지 못하는 치명적인 결함이 있었습니다.
- **현재 상태**: 프론트엔드의 복잡한 통신 로직을 모두 걷어내고, 백엔드 팀이 설계한 단일 채팅 엔드포인트(`POST /api/ai-agent/chat`)만 바라보도록 완벽하게 리팩터링했습니다.
- **결과**: `ai-search-bar.tsx` 내부에 덕지덕지 붙어있던 불필요한 서비스 호출(`storeService`, `portfolioService`)이 삭제되어 코드가 매우 깔끔해졌습니다.

### ✅ 응답 구조에 맞춘 UI 렌더링 준비
- 백엔드가 응답해 주는 `actionType` (`PORTFOLIO_LIST`, `SHOW_SCHEMA` 등)에 따라 UI가 조건부로 렌더링되도록 로직 스위칭을 구현해 두었습니다.

### ✅ 원상 복구 (Clean Code)
- 로컬 테스트를 위해 임시로 삽입했던 가짜(Mock) 데이터와 백엔드의 강제 인증 해제 코드를 **모두 100% 제거**하고, 실제 배포 서버를 바라보는 순정 상태로 완벽히 원상 복구했습니다.

## 2. 현재 직면한 문제 (Blockers) 🚧

현재 작성된 프론트엔드 오케스트레이터 코드를 실제로 구동(Test)해 보려면 백엔드와 통신해야 하는데, 다음 두 가지 병목이 존재합니다.

### 🔴 Blocker 1: 백엔드 인증(Auth) 장벽 (401 Unauthorized)
- **상황**: 백엔드의 `/api/ai-agent/chat` API에는 스프링 시큐리티의 `@PreAuthorize("isAuthenticated()")`가 걸려있어 인증 토큰(JWT)을 요구합니다.
- **문제**: 하지만 프론트엔드 프로젝트에는 아직 실제 로그인 기능(토큰 발급 로직)이 연동되어 있지 않습니다.
- **해결 방안**: 프론트엔드 로그인 연동을 우선적으로 개발하거나, 백엔드 개발자 측에서 해당 API의 인증을 임시로 해제해 주어야 합니다.

### 🔴 Blocker 2: 백엔드 로컬 실행 환경(자바 버전) 충돌
- **상황**: 백엔드 서버를 로컬에서 직접 띄워 우회 테스트를 진행하려 했습니다.
- **문제**: 유저님의 PC 환경(Java 25)과 백엔드 프로젝트의 구형 빌드 도구(Gradle 8.14 및 Lombok) 버전이 호환되지 않아 로컬 서버 빌드가 불가능한 상태입니다.
- **해결 방안**: 로컬에 의존하지 않도록 백엔드 팀에서 `Docker(도커)` 환경을 제공해 주거나, VSCode 대신 인텔리제이(IntelliJ)를 활용해 독립적인 JDK(Java 21) 환경에서 실행해야 합니다.

## 3. 향후 계획 (Next Steps) 🎯

현재 통신 테스트가 막혀있으므로, **"환경과 무관하게 진행할 수 있는 프론트엔드 독립 작업"**을 먼저 치고 나가는 것을 제안합니다.

1. **주문서 폼(OrderSchemaForm) UI 고도화**:
   - 백엔드가 완성된 주문 스키마(Slots)를 넘겨주었을 때, 이를 예쁘게 띄워줄 수 있는 동적 모달 폼 컴포넌트(`OrderSchemaForm.tsx`)의 디자인 및 로직을 고도화합니다.
2. **로그인/인증 시스템 연동 준비**:
   - 나중에 JWT 토큰을 발급받았을 때 이를 안정적으로 관리하고(AsyncStorage), `ai.ts` 통신 시 헤더에 `Authorization`을 주입하는 인프라 코드를 미리 작성해 둡니다.
3. **백엔드/데브옵스 협업 대기**:
   - 백엔드 팀이 Auth 문제를 해결해주거나 Docker 환경을 제공해 줄 때까지 대기합니다.
