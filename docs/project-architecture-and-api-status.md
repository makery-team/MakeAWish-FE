# MakeAWish Frontend 프로젝트 아키텍처 및 API 연동 현황 (2026-05-27 기준)

본 문서는 현재 MakeAWish 프론트엔드(React Native / Expo) 프로젝트의 **폴더/파일 구조와 각 역할**, 그리고 노션(Notion) 명세서 기준 **API 연동 진행률**을 명확히 파악하기 위해 작성되었습니다.

---

## 1. 프론트엔드 프로젝트 폴더 및 파일 구조 (Architecture)

MakeAWish 프론트엔드는 관심사 분리(Separation of Concerns) 원칙에 따라 UI 로직과 통신 로직을 명확하게 나누어 관리하고 있습니다.

```text
MakeAWish-FE/
├── app/                  # 📍 [라우팅 & 페이지] Expo Router 기반의 화면(Screen) 파일들
│   ├── (tabs)/           # 탭 네비게이션 화면들 (홈, 탐색, 찜, 마이페이지 등)
│   └── _layout.tsx       # 전체 앱의 레이아웃 및 테마 설정
│
├── components/           # 🧩 [UI 컴포넌트] 화면을 구성하는 재사용 가능한 레고 블록들
│   ├── ai-search-bar.tsx # [핵심] AI 챗봇 UI 및 오케스트레이션 결과물(영수증, 결제 창)을 렌더링하는 컴포넌트
│   └── image-slider.tsx  # 포트폴리오 이미지 슬라이더 (시안 편집하기, 문의하기 버튼 포함)
│
├── services/             # 📡 [API 통신 계층] 백엔드(Spring Boot) 서버와의 네트워크 통신 전담
│   ├── ai.ts             # AI 채팅 통신 (`/api/ai-agent/chat`) 및 인페인팅(`inpaint`) 통신
│   ├── feed.ts           # 피드 조회 및 추천 로직
│   ├── portfolio.ts      # 포트폴리오 검색 및 태그 기반 통신
│   └── store.ts          # 스토어 관련 로직
│
├── types/                # 🏷️ [타입 정의] TypeScript 인터페이스 모음 (데이터 구조 명세서)
│   └── index.ts          # Message 구조 (actionType, slots 등 백엔드 지시어 포함)
│
├── docs/                 # 📚 [문서] 개발 일지 및 아키텍처 가이드 문서 보관소
│
└── package.json          # 📦 프로젝트 패키지 및 라이브러리(expo-linear-gradient 등) 관리 파일
```

### 💡 구조 설계 핵심 원칙 (Clean Architecture)
- **`components/`는 멍청해야 합니다 (Dumb Components):** 이곳의 파일들은 스스로 DB를 조회하거나 판단하지 않고, 오직 데이터(`actionType`)를 받아 화면을 예쁘게 그리는 데에만 집중합니다.
- **모든 판단은 백엔드가 담당합니다:** 프론트엔드는 유저의 입력을 `services/ai.ts`를 통해 백엔드로 넘기고, 백엔드가 내린 지시사항(`CONFIRM_SLOTS` 등)에 맞춰 화면만 전환합니다. (Backend Orchestration)

## 2. API 연동 진행 현황 (API Integration Status)

### ✅ 2.1 연동 완료 및 고도화 중 (진척도 높음)
이 영역은 프론트엔드 코드베이스에 통신 로직과 UI가 모두 자리를 잡은 핵심 기능들입니다.

*   **[홈/탐색] 태그 기반 피드 조회 (`GET /api/portfolios/feeds`)**
    *   **상태:** 🟢 완료
    *   **내용:** 태그 기반 피드 불러오기 및 무한 스크롤(Infinite Scroll) UI 연동 완료
*   **[주문] 주문서 스키마 조회 (`GET /api/stores/{storeId}/order-schema`)**
    *   **상태:** 🟢 완료 (AI 오케스트레이션 기반)
    *   **내용:** 단순 GET 조회를 넘어, AI 채팅을 통해 백엔드로부터 `SHOW_SCHEMA` 액션을 받아 동적으로 주문서를 렌더링하도록 퀄리티를 높였습니다.
*   **[디자인] 포트폴리오 인페인팅 생성 (`POST /api/portfolios/{portfolioId}/inpaintings`)**
    *   **상태:** 🟡 진행 중 (메서드 구현 완료)
    *   **내용:** `services/ai.ts` 내에 통신 메서드를 신규 추가하여 언제든 호출할 수 있는 상태입니다.

---

### 🚧 2.2 뼈대(UI)는 완료, 실제 API 통신은 대기 중
이 영역은 UI 프로토타입은 모두 구현되어 있으나, 아직 버튼 클릭 시 실제 API를 쏘는 로직을 연결하지 않은 곳입니다.

*   **[주문] 주문 생성 (`POST /api/orders`) 및 결제 연동**
    *   **상태:** 🟡 진행 중 (UI 완료)
    *   **내용:** `CONFIRM_SLOTS`(주문 확인 영수증)와 `ORDER_SUMMARY`(결제 금액 카드) UI가 프로토타입 시안대로 100% 반영되었습니다. '결제하기' 버튼에 POST 통신만 붙이면 완료됩니다.
*   **[인증] 회원가입/로그인 및 토큰 재발급**
    *   **상태:** 🟡 진행 중 (백엔드 세팅 대기)
    *   **내용:** OAuth2 로그인 흐름의 프론트엔드 로직은 존재하나, 백엔드의 Spring Security 및 Docker 환경 세팅이 완료되어야 정상 연동 테스트가 가능합니다. (현재 401 에러 우회 상태)

---

### ⏸️ 2.3 미구현 영역 (시작 전)
아직 프론트엔드 작업이 들어가지 않은, 백엔드 명세서 상의 추가 기능들입니다.

*   **[상세 페이지]** 매장 상세 정보, 리뷰, 매장 포트폴리오 조회
*   **[마이페이지]** 내 정보 조회/수정, 알림 조회, 포트폴리오 찜하기/취소, 내 댓글 관리 등
*   **[채팅]** 실시간 통신(WS) 기반의 유저-매장 간 1:1 일반 채팅 (현재의 AI 챗봇과는 별개 시스템)

---

### 🚀 넥스트 스텝 제안
1. **주문/결제 플로우 완성:** 현재 만들어진 프로토타입 결제 UI의 **[결제하기]** 버튼에 `POST /api/orders` 를 붙여 실제 주문이 들어가는지 테스트해 봅니다.
2. **상세 페이지 뼈대 잡기:** 아직 시작하지 않은 매장 상세 페이지(`GET /api/stores/{storeId}`)의 UI를 디자인하고 데이터를 받아오는 로직을 작성합니다.
