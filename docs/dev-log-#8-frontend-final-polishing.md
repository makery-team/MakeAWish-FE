# Dev Log #8: 시연 전 프론트엔드 UI/UX 최종 폴리싱 및 트러블슈팅

## 🎯 개요
이번 개발 로그에서는 백엔드 API와의 최종 연동 과정 및 앱 사용성 테스트 중 발견된 자잘하지만 크리티컬한 UI/UX 버그들을 해결한 과정과 로직을 상세히 기록합니다. 특히 토큰 만료 문제, 비동기 로딩 화면 깜빡임, 라우팅 스택 꼬임 등 리액트 네이티브(React Native) 주니어 개발자들이 자주 겪을 수 있는 트러블슈팅 사례를 집중적으로 다룹니다.

---

## 🛠 작업 내용 및 트러블슈팅 (Troubleshooting & Logic)

### 1. 전역 세션(Auth) 및 토큰 만료(401) 자동 로그아웃 처리
- **현상**: 로그인한 지 오래된 유저의 경우 `accessToken`이 만료되어 `401 Unauthorized` 에러가 발생합니다. 문제는 구버전 앱에서 로그인한 유저들의 로컬 스토리지(`AsyncStorage`)에 연장용 토큰인 `refreshToken`이 없어 무한 통신 에러 루프에 빠진 채로 화면이 굳어버리는 버그가 있었습니다.
- **해결 로직**: 
  - `utils/api.ts`의 `fetchWithAuth` 유틸리티 함수 내부에 토큰 갱신 실패(`refreshAccessToken() == null`) 조건문을 추가했습니다.
  - 순수 JS/TS 파일에서는 훅(`useAuth()`)이나 네비게이션(`useRouter()`)을 사용할 수 없으므로, `DeviceEventEmitter.emit('EXPIRED_SESSION')`을 호출하여 앱 전역에 알림(Event)을 쏘는 방식을 채택했습니다.
  - 최상단 컨텍스트인 `context/AuthContext.tsx`의 `useEffect` 안에서 해당 이벤트를 리스닝하고, `signOut()` 상태 초기화 및 `Alert` 알림과 동시에 로그인 창으로 튕겨나가도록 안전망(Safety Net)을 구현했습니다.

### 2. 라우팅 흐름 완벽 제어 (Navigation Stack Management)
- **현상**: 매장 상세 페이지 또는 AI 캔버스 에디터 화면에서 "이 시안으로 상담하기" 버튼을 눌렀을 때, 유저의 기대와 달리 AI 채팅 화면(홈)으로 가지 않고 이전 페이지(상세페이지)로 돌아가는 오류가 있었습니다.
- **원인**: Expo Router의 `router.back()` 메서드는 네비게이션 스택에 쌓여있던 직전 히스토리로 무조건 팝(Pop)하기 때문에, `상세페이지 -> 에디터`로 진입한 유저는 `뒤로가기` 시 홈 탭이 아닌 상세페이지 탭으로 돌아가버립니다.
- **해결 로직**: 
  - `app/editor/[id].tsx` 및 `app/shop/[id].tsx`의 라우팅 콜백 함수(`handleInquiry`)를 수정했습니다.
  - 현재 쌓인 네비게이션 스택을 갈아끼우며 최상단 메인 탭으로 강제 다이렉트 이동하는 `router.replace('/(tabs)')` 메서드로 교체하여 자연스러운 뎁스(Depth) 흐름을 보장했습니다.

### 3. 비동기 렌더링 시 깜빡임 현상 완화 (Loading State UX)
- **현상**: 지도에서 매장을 누르고 상세 페이지로 들어갈 때, 1초 정도 빈 화면에 "매장 정보를 찾을 수 없습니다"라는 에러 문구가 떴다가 뒤늦게 정보가 렌더링되는 현상이 있었습니다.
- **원인**: 백엔드 API에서 `storeData`를 비동기로 받아오기 전의 초기값은 `null`입니다. React의 빠른 렌더링 사이클이 로딩 중인지 여부를 판단하지 않고 즉시 예외 처리 화면을 반환했습니다.
- **해결 로직**: 
  - 상태 변수 `isLoadingDetail`의 조건부 렌더링을 1순위로 끌어올렸습니다. 데이터가 도착할 때까지는 예외 화면이 아닌 `ActivityIndicator` (빙글빙글 도는 로딩 스피너)를 렌더링하도록 `components/shop-detail.tsx` 코드를 수정했습니다.

### 4. 기능 미구현에 따른 프론트엔드 방어 로직 (Feature Toggling)
- **현상**: 매장 상세의 "1:1 채팅하기" 버튼을 누르면 `WebSocket 1006 Closed` 에러가 발생하며 서버 연결이 튕겼습니다.
- **원인**: 1:1 채팅 방 생성 로직(`makeRoom`)에는 판매자의 `sellerId`가 필수로 요구되지만, 현재 릴리즈된 `StoreResponse` 백엔드 DTO에는 이 값이 포함되어 있지 않습니다.
- **해결 로직**: 
  - 런타임 크래시를 유발하는 죽은 코드(Dead Code)나 미완성 API 연동부는 시연 및 앱 안전성을 저해합니다.
  - 따라서, 해당 라우터 푸시 로직을 임시 주석 처리하고 `Alert.alert('안내', '현재 점검 중입니다.')`를 띄우도록 방어적인 처리를 마쳤습니다.

### 5. 지도 마커 버그 및 데이터 바인딩
- **현상**: `CAKE_SHOPS` 모의(Mock) 데이터를 제거한 뒤 안드로이드에서 커스텀 마커가 무한 리렌더링되거나 탭(클릭) 이벤트가 씹히는 문제 발견.
- **해결 로직**: 안드로이드 맵 오버레이 마커 렌더링 최적화를 진행했으며, 백엔드로부터 응답받은 `StoreResponse`의 `categories` 내부 배열을 평탄화(FlatMap)하여 갤러리에 포트폴리오를 동적 주입했습니다.

## 📝 주니어 개발자(Junior Devs)를 위한 교훈 요약
1. **이벤트 리스너 활용**: 훅(Hooks) 사용이 불가능한 일반 TypeScript 환경에서 컴포넌트로 신호를 보낼 땐 `DeviceEventEmitter` 패턴을 고려해보세요.
2. **비동기 방어 코딩**: 네트워크 비용에 의해 발생하는 지연(Latency) 시간을 잊지 마세요. 스피너나 스켈레톤(Skeleton) UI 처리는 UX의 핵심입니다.
3. **네비게이션 스택 이해**: 무지성 `back()`보다 `push()`와 `replace()`의 차이를 명확히 인지하고 유저 여정(User Journey)을 설계하세요.
4. **점진적 런칭 전략**: 서버 API가 아직 준비되지 않았다면 프론트엔드에서 고장난 기능을 그대로 열어두지 말고 우아하게 알림(Graceful Degradation)으로 막는 센스가 필요합니다.
